# MiniCC (Mini Claude Code) 项目总体规划文档

## 1. 项目愿景与技术选型
- **项目名称**：MiniCC - 极简工程级智能编程助手
- **核心定位**：基于架构图的 8 大模块，实现一个可在浏览器中运行、具备完整“计划-执行-审批”闭环的 AI 运维/开发助手。
- **后端 (The Engine)**：Python 3.14.6 + FastAPI + WebSocket + Pydantic V2。
- **前端 (The Shell)**：Next.js 14+ (App Router) + React Server Components + TailwindCSS + Shadcn/ui。
- **通信协议**：RESTful (任务创建/历史查询) + WebSocket (主循环流式输出、工具调用实时回调)。

## 3. 开发阶段与里程碑 (Milestones)

我们将整个开发过程分为 **5 个阶段**，采用敏捷迭代。

### Phase 0: 基础骨架与协议定义 (Foundation)
**目标**：跑通 FastAPI + Next.js 联调，定义所有数据契约。

- **任务 0.1**：搭建 Backend 项目结构，配置 `pyproject.toml` (依赖 FastAPI, Uvicorn, Pydantic, HTTPX, PyGithub/GitPython)。
- **任务 0.2**：定义核心 Pydantic 模型 (统一协议层)：
  - `ToolCall`, `ToolResult`, `ContentBlock` (Text/Image/File)。
  - `SessionState`, `MessageHistory`, `PermissionRequest`。
- **任务 0.3**：搭建 Next.js 项目，配置 Tailwind + Shadcn，实现静态布局（左侧历史、右侧对话流）。
- **任务 0.4**：建立 WebSocket 连接心跳机制，确保前后端长连接稳定。

---

### Phase 1: 会话主循环与上下文装配 (The Core Loop)
**目标**：实现“模型调用 -> 工具调用 -> 结果回流”的闭环，能看懂项目。

- **任务 1.1**：实现 `ContextBuilder` (模块 4)。
  - 扫描当前工作目录 (Workspace) 的 `.git` 状态。
  - 读取 `.minicc/rules.md` 和 `memory.md` (项目规则与长期记忆)。
  - 注入当前系统日期、OS 信息。
- **任务 1.2**：实现轻量级 `QueryEngine` (模块 2)。
  - 设计提示词模板 (System Prompt + User Query)。
  - 集成 OpenAI/Anthropic SDK (可配置本地 Ollama 作为备选)。
  - 处理流式响应 (Streaming) 并转发至前端。
- **任务 1.3**：实现工具调用的解析器 (Tool Call Parser)，区分 `native` 函数调用 和 `XML`/`JSON` 模式。
- **任务 1.4**：前端渲染流式消息，支持 Markdown 实时语法高亮。

---

### Phase 2: 真实工具落地与权限系统 (The Action Layer)
**目标**：让 AI 真正能读写文件、执行命令，但必须“先审批后执行”。

- **任务 2.1**：实现 `FileSystem` 工具集 (模块 6)。
  - `read_file`: 带行号显示。
  - `write_to_file` / `str_replace_editor`: 精细化的代码编辑 (支持 Diff 预览)。
  - 引入 `path_validator` 防止越权访问 (沙箱限制在当前项目根目录)。
- **任务 2.2**：实现 `ShellExecutor` 工具集 (模块 6)。
  - 支持交互式命令 (需携带 `pty` 伪终端)。
  - 实现超时控制 (Timeout) 和输出截断 (防止炸屏)。
- **任务 2.3**：实现 `Permission System` (模块 5)。
  - 定义权限等级：`READ` (自动允许), `WRITE` (需审批), `EXECUTE` (需严格审批)。
  - 实现“审批中间件”：工具被调用时，主循环挂起 (Wait)，通过 WebSocket 推送 `PermissionRequest` 到前端。
- **任务 2.4**：前端开发“工具调用卡片” (Tool Card)：
  - 显示 `diff` 变更内容。
  - 显示 Shell 命令详情。
  - 提供 `Approve`, `Reject`, `Approve & Always Allow` 三个按钮。

---

### Phase 3: 状态持久化与任务系统 (State & Task)
**目标**：支持长任务、后台运行、会话断开重连后状态不丢失。

- **任务 3.1**：设计 `Session State` 存储方案 (模块 7)。
  - 使用 Redis (存热数据) + SQLite/Postgres (存历史记录)。
  - 序列化整个 `Message History` 和 `Tool Call Stack`。
- **任务 3.2**：实现 `TaskManager`。
  - 支持 `async` 后台任务。当用户关闭浏览器，后端 Agent 继续运行 (Background Task)。
  - 实现“恢复会话”功能：用户重连时，后端重放 (Replay) 最近的 5 条消息记录给前端。
- **任务 3.3**：实现中断控制 (Interrupt/Cancel)。
  - 前端点击“停止生成”时，通过 WebSocket 发送 `CancelSignal`，后端利用 Python 3.14 的 `TaskGroup` 取消正在运行的异步工具。

---

### Phase 4: 扩展系统与平台化 (Extensions)
**目标**：接入 MCP 和 LSP，为接入第三方生态做准备。

- **任务 4.1**：实现 MCP Client (模块 8)。
  - 启动子进程 (Stdio) 或连接 SSE 服务器。
  - 将 MCP 暴露的 `Tools` 动态注册到主循环的 `ToolRegistry` 中。
- **任务 4.2**：实现 LSP 集成 (Language Server Protocol)。
  - 启动 Python/TS 的 LSP 服务器，提供 `go_to_definition`, `find_references` 等工具，增强 AI 对代码库的理解（即使不直接调用，也可作为上下文增强）。
- **任务 4.3**：插件热加载机制。
  - 扫描 `~/.minicc/plugins` 目录，动态导入 Python 文件中的 `Tool` 子类。

---

### Phase 5: 打磨与优化 (Polish)
- **任务 5.1**：Token 预算管理 (Budget Manager)，防止大上下文撑爆 Context Window。
- **任务 5.2**：UI/UX 优化，适配暗黑模式，为工具调用添加加载骨架屏。
- **任务 5.3**：打包分发 (Docker 镜像 或 PyInstaller + Next.js 静态导出)。

## 4. 关键技术选型难点与对策

| 难点 | 架构映射 | 解决方案 (MiniCC 策略) |
| :--- | :--- | :--- |
| **Python 异步与 Shell 交互** | 模块 6 | 使用 `asyncio.create_subprocess_exec` 配合 `websockets` 实时流式输出命令结果。 |
| **工具输入格式统一** | 模块 3 | 强制所有 Tool 继承 `BaseTool`，输入输出必须为 Pydantic `BaseModel`，前端据此自动生成表单。 |
| **大文件编辑 Diff** | 模块 5 + UI | 后端利用 `difflib` 生成 Unified Diff，前端使用 `react-diff-view` 渲染。 |
| **多 Agent 协同** | 模块 7 (扩展) | 初期仅支持单 Agent 主循环，后期使用 `Sub-Agent` 作为特殊的 Tool 调用（子任务隔离）。 |
| **Next.js 与长连接** | 模块 1 | 使用 Next.js `App Router` 的 `route.ts` 处理 WebSocket 升级，或将 WebSocket 独立为 Node.js Server，通过 Redis 与 FastAPI 同步。*（建议：FastAPI 直接挂载 WebSocket 端点，Next.js 仅作为静态客户端）* |

## 5. 核心数据流 (MiniCC 闭环)

为了帮你更好地编写代码，这里梳理一下最关键的用户操作数据流：

1. **用户输入** -> Next.js (`useWebSocket`) -> `ws://backend/chat/{session_id}`
2. **Backend QueryEngine** -> 调用 `ContextBuilder` 拼装 Prompt -> 请求 LLM。
3. **LLM 返回 Tool Call** (e.g., `write_to_file`) -> `QueryEngine` 挂起流程，调用 `PermissionSystem`。
4. **Permission System** -> 通过 WebSocket 下发 `{"type":"permission_required", "tool":"write_to_file", "content":"..."}`。
5. **前端 UI** -> 亮起红色审批卡片，用户点击 "Approve" -> 发送 `{"action":"approve", "id":"..."}`。
6. **Backend** -> 权限通过，执行实际的 `FileSystem` 操作 -> 结果追加到消息历史 -> 继续请求 LLM（观察结果）。
7. **LLM** -> 生成最终文本回复 -> WebSocket 推送 `message_chunk` -> 前端渲染。
