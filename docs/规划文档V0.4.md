# MiniCC V0.4 — AI-Native 集成开发环境

> **一句话定位**：V0.1 是"能写代码的 AI"，V0.2 是"能操作一切数字系统的 AI"，V0.3 是"能编排复杂工作流的 AI 平台"。  
> **V0.4 是"AI 原生集成开发环境"** — 不只是聊天窗，而是一个完整的、AI 为第一公民的软件开发环境。

---

## 1. 为什么是 AI-Native IDE？

当前 AI 编码工具的格局：

| 工具 | 方式 | 局限性 |
|:-----|:-----|:--------|
| **Cursor** | AI 增强的 VS Code | 闭源、依赖云端、数据隐私 |
| **GitHub Copilot** | IDE 插件 | 仅补全/聊天，不能全盘控制 |
| **Claude Code** | 终端 Agent | 无 GUI、无工作流、无 RPA |
| **Cline** | VS Code 插件 + Agent | 依赖 VS Code 生态 |
| **Aider** | 终端 Agent | 纯 CLI，无 GUI |
| **MiniCC V0.1-0.3** | Agent + RPA + Graph | 无原生编辑体验 |

**V0.4 的差异化定位**：  
> 一个**自托管、AI 原生、模块化**的开发环境，不是 IDE 插件，而是 IDE 本身。  
> AI 不是辅助工具，而是开发流程中的**平等协作者**。

## 3. 关键技术决策

### 3.1 编辑器引擎：Monaco Editor

```text
Monaco Editor (VS Code 同源)
├── 优点：完全浏览器端、语法高亮、智能提示、Diff 编辑器
├── 集成方式：Next.js iframe / Webpack 插件
├── AI 编辑协议：自定义 LSP 扩展 (mini-lsp)
└── 文件系统：Browser FS + 后端 WebDAV 协议
```

### 3.2 AI 编辑协议

不再使用 `str_replace_editor` / `write_to_file` 等工具进行编辑，而是设计**原生编辑器协议**：

```python
# AI 编辑器协议 — 直接操作编辑器状态
class EditorAction:
    select: Range          # 选择范围（行/列）
    replace: str           # 替换内容
    cursor: Position       # 光标位置
    explanation: str       # AI 的解释

# 流式编辑（AI 逐字符输出到编辑器）
class StreamEdit:
    position: Position
    characters: str        # 逐字符流
    finalize: bool         # 是否完成
```

### 3.3 多 Agent 协作

```
User Prompt
  │
  ├── Architect Agent（架构师）
  │   └── 设计方案 → 拆分子任务
  │
  ├── Coding Agent（编码员）[多个并行]
  │   ├── Agent A: 实现模块 A
  │   ├── Agent B: 实现模块 B
  │   └── Agent C: 编写测试
  │
  ├── Reviewer Agent（审查员）
  │   └── Code Review → 反馈修改
  │
  └── Tester Agent（测试员）
      └── 运行测试 → 报告结果
```

### 3.4 编辑器工具（替代 V0.1 文件工具）

| V0.1 工具 | V0.4 替代 | 优势 |
|:----------|:----------|:-----|
| `read_file` | 编辑器打开文件 | 语法高亮、折叠、实时 |
| `write_to_file` | AI 流式写入 | 逐字符显示、可撤销 |
| `str_replace_editor` | AI Select+Replace | 可视选择、精准定位 |
| `glob` | 文件树浏览 | 目录树、搜索 |
| `grep` | 编辑器搜索面板 | 上下文预览、替换 |

## 4. V0.4 开发阶段

### Phase M：编辑器引擎（4 周）
**目标**：在浏览器中运行 Monaco Editor，文件系统与后端同步。

| 任务 | 说明 |
|:-----|:------|
| M1 | Monaco Editor 集成到 Next.js |
| M2 | 文件浏览器组件（树形 + 语义搜索） |
| M3 | 后端文件同步协议（WebDAV / WebSocket FS） |
| M4 | 多 Tab 编辑（对比 V0.3 多对话 Tab） |
| M5 | Diff 编辑器（AI 变更可视化） |

### Phase N：AI 原生编辑（3 周）
**目标**：AI 能直接操作编辑器，而不是通过文件工具。

| 任务 | 说明 |
|:-----|:------|
| N1 | AI 编辑协议（Select + Replace + Stream） |
| N2 | Tab-to-Accept（内联代码补全） |
| N3 | Natural Edit（自然语言选 → 改） |
| N4 | Multi-cursor AI 批量操作 |
| N5 | Undo/Redo 支持 AI 操作 |

### Phase O：多 Agent 协作（3 周）
**目标**：多个 AI Agent 并行协作开发。

| 任务 | 说明 |
|:-----|:------|
| O1 | Agent 会话隔离（每个 Agent 独立上下文） |
| O2 | 架构师 → 编码员 → 审查员流水线 |
| O3 | 自动 Git 工作流（分支/PR/Merge） |
| O4 | 冲突解决（并行编辑冲突处理） |
| O5 | 项目级长期记忆 |

### Phase P：可视化 Debugger（2 周）
**目标**：AI 能实时观察代码运行状态。

| 任务 | 说明 |
|:-----|:------|
| P1 | 集成 Python Debugger (pydebug) |
| P2 | 变量监视面板（AI 可读） |
| P3 | 调用栈可视化 |
| P4 | AI 自动 Debug 循环（断点 → 检查 → 修复） |

### Phase Q：部署与生态（3 周）
**目标**：一键部署、桌面端、插件市场。

| 任务 | 说明 |
|:-----|:------|
| Q1 | Docker Compose 一键部署（含 Chromium） |
| Q2 | Tauri 桌面端封装 |
| Q3 | 插件/技能市场 |
| Q4 | WebSocket 实时协作（多用户编辑） |
| Q5 | CI/CD 集成（GitHub Actions → Auto Deploy） |

## 8. 风险与缓解

| 风险 | 缓解 |
|:-----|:------|
| **Monaco 集成复杂度高** | 参考 Cursor 的开源方案，分步实现 |
| **多 Agent 成本高** | 共享上下文、按需创建、token 预算控制 |
| **与 Cursor 直接竞争** | 差异化在自托管 + 开源 + RPA + Graph |
| **前端性能** | Web Worker + 虚拟滚动 + 增量渲染 |
| **LLM 延迟** | 流式编辑、提前预测、并行生成 |
