## 1. 任务目标

实现工具调用解析器，能够从 LLM 返回的流式输出中正确识别并提取工具调用，支持多种格式（Anthropic tool_use block、OpenAI function_call、以及可选的 XML/JSON 文本模式），并将解析结果路由到对应的 Tool 执行器。

### 2.2 支持的解析格式

#### 格式 A：Anthropic native tool_use (首选)

```json
{
  "type": "tool_use",
  "id": "toolu_abc123",
  "name": "read_file",
  "input": {"path": "src/main.py", "offset": 0, "limit": 100}
}
```

- [ ] 直接从 `content` 列表中提取 `type == "tool_use"` 的 block
- [ ] `id` 由 Anthropic API 自动生成

#### 格式 B：OpenAI function_call

```json
{
  "function": {
    "name": "read_file",
    "arguments": "{\"path\": \"src/main.py\"}"
  }
}
```

- [ ] `arguments` 是 JSON 字符串，需要二次解析
- [ ] 需要自行生成 `id`

#### 格式 C：XML 兼容模式（备用）

```xml
<invoke name="read_file">
<parameter name="path">src/main.py</parameter>
</invoke>
```

- [ ] 用 `xml.etree.ElementTree` 解析
- [ ] 为不支持原生 tool_use 的弱模型提供备选

#### 格式 D：JSON 文本模式（备用）

````
```tool_call
{
  "name": "read_file",
  "input": {"path": "src/main.py"}
}
```
````

- [ ] 用正则匹配代码块中的 `tool_call` 标记

### 2.3 工具路由系统 (`ToolRouter`)

- [ ] 文件：`backend/app/core/parser.py`

```python
class ToolRouter:
    """
    工具路由系统。
    将解析出的 ToolCall 路由到对应的 Tool 执行器，
    并在执行前后触发权限检查、结果回传等钩子。
    """
    
    def __init__(self, registry: ToolRegistry):
        self.registry = registry
    
    async def route(
        self,
        tool_call: ToolCall,
        permission_callback: Callable,
    ) -> ToolResult:
        """路由一个工具调用"""
        # 1. 查找工具
        tool = self.registry.get(tool_call.name)
        if not tool:
            return ToolResult(
                tool_call_id=tool_call.id,
                output=f"错误：未知工具 '{tool_call.name}'",
                is_error=True
            )
        
        # 2. 验证输入
        try:
            validated_input = tool.input_schema.model_validate(tool_call.input)
        except ValidationError as e:
            return ToolResult(
                tool_call_id=tool_call.id,
                output=f"输入验证失败: {e}",
                is_error=True
            )
        
        # 3. 权限检查 (Phase 2)
        # permission_callback(tool_call, tool.permission_level)
        
        # 4. 执行
        result = await tool.execute(validated_input)
        return result
```

### 2.6 与 QueryEngine 集成

- [ ] QueryEngine 在收到 LLM 流式事件后：
  - 若为 `tool_use` block → 调用 `ToolCallParser.parse_anthropic_tool_use()`
  - 若为文本流 → 累积并尝试检测 XML/JSON 模式
- [ ] 解析出的 ToolCall 进入路由流程
- [ ] 解析失败 → 给 LLM 返回错误消息，让 LLM 自行修正

## 5. 注意事项

- 优先支持 Anthropic native tool_use，这是主要格式
- XML/JSON 文本模式是备选，仅在模型不支持原生 tool_use 时启用
- 解析器必须健壮：面对格式错误的输入不能崩溃，应返回有意义的错误
- 支持流式场景：部分 tool_use block 可能跨多个 chunk（Anthropic 不拆分 tool_use，但其他 provider 可能拆）
